# 前端设计规范

职责：本文是前端产品气质、视觉规范、信息架构、组件原则、设计 token 和可访问性要求的主要来源。

职责边界：

- 本文记录“界面应该长什么样”和“哪些视觉/交互模式禁止使用”。
- 后续路线和功能矩阵写入 `PLAN.md`，当前状态和验证写入 `PROGRESS.md`。
- API adapter、类型、状态、错误处理、依赖、本地开发、CI 和部署规则写入 `IMPLEMENTATION_GUIDE.md`。
- 测试策略和发布门禁分别写入 `TESTING.md` 与 `CHECKLIST.md`；本文只记录设计和可访问性要求，不维护完整 checklist。
- 风险登记写入 `RISKS.md`，设计灵感和外部资料写入 `REFERENCES.md`。

后续修改 React + shadcn/ui 页面时，必须先读取本文档。

## 产品气质

SkyRadar 是一个 GitHub 开源项目发现系统，面向安全团队、研发负责人和工程管理者。

界面应当：

- 克制、专业、可信。
- 信息密度高，但不拥挤。
- 强调发现、状态、筛选、排查和处理效率。
- 像安全运营控制台，而不是营销落地页。
- 让用户能快速回答：发现了哪些新项目、项目属于哪个规则、是否已处理、下一步该做什么。

界面不应当：

- 使用大面积渐变、装饰光晕、玻璃拟态或抽象背景。
- 使用 oversized hero、营销式首屏或欢迎页。
- 使用过多紫色、蓝紫渐变或常见 AI 生成风格。
- 使用卡片套卡片。
- 为了视觉效果牺牲表格扫描效率。
- 用说明文字解释界面怎么用；控件应当自解释。

## 视觉方向

审美方向：高信息密度的安全运营后台。

参考关键词：

- 开源项目发现控制台
- 数据密集型后台
- 克制的企业工具
- 清晰处理状态
- 可扫描表格
- 低噪音设置页

可借鉴的产品气质：

- Linear 的克制层级和精确间距。
- GitHub 的信息密度和朴素控件。
- Vercel 的留白控制和中性色使用。
- 云安全控制台的风险状态表达。

这些是方向参考，不要求复制任何品牌视觉。

## 信息架构

首屏应直接呈现可操作工作台：

- 顶部展示关键统计和任务状态。
- 中部提供筛选和结果表格。
- 详情页围绕风险判断和排查行动组织。
- 设置页按任务、规则、账号、通知等工作流分区。

页面不需要独立 landing、hero、介绍区或营销文案。

## 当前实现结构

- `client/src/App.tsx` 定义应用路由：`/`、`/view/leakage/:id`、`/setting`、`/setting/:tab`，未匹配路由重定向到结果页。
- `client/src/components/layout/AppShell.tsx` 提供统一顶栏、SkyRadar 品牌、主导航和最大 `1440px` 内容容器。
- `client/src/pages/ResultsPage.tsx`、`LeakageDetailPage.tsx`、`SettingsPage.tsx` 做页面级编排。
- `client/src/features/results/*` 承载结果仪表盘、筛选、表格、查询状态和格式化。
- `client/src/features/settings/*` 承载 GitHub 账号、查询规则、任务调度、黑名单、通知、SMTP 和 webhook 设置。
- `client/src/components/ui/*` 是通用 UI primitives，页面业务逻辑不得写入此目录。
- `client/src/lib/api/*` 负责 `/api/v1/*` endpoint、请求参数序列化、REST response envelope 适配和敏感字段剔除。
- `client/src/types/api.ts` 是共享 API 类型边界。
- `client/src/index.css` 是 Tailwind v4 主题变量和当前设计 token 的实现源。

## 颜色原则

默认使用中性色承载大部分界面，语义色只用于状态表达。

背景：

- 页面背景：`#ffffff`
- 主内容面：`#ffffff`
- 次级内容面：`#f6f8fa`
- 边框：`#d1d9e0`
- 分割线：`#d1d9e0`

文本：

- 主文本：`#1f2328`
- 次级文本：`#59636e`
- 弱文本：`#6e7781`

语义色：

- 风险：`#cf222e`
- 警告：`#bf8700`
- 安全：`#1a7f37`
- 信息：`#0969da`

规则：

- 风险色只用于真实风险、涉密、失败和危险操作。
- 安全色只用于已确认安全、成功和健康状态。
- 不使用彩虹状态色。
- 不使用纯黑 `#000000` 作为正文。
- 不让单一色相统治整个界面。

## 布局原则

- 后台页面优先 12 栅格或弹性网格。
- 主要内容宽度应充分利用桌面空间。
- 表格和过滤器优先水平扫描效率。
- 统计卡片数量少而准，不做装饰性堆叠。
- 设置页使用清晰分区，不做卡片套卡片。
- 移动端允许纵向堆叠，但不能遮挡、重叠或横向溢出。

## 组件原则

- 按钮用于明确命令，危险操作需要明确视觉区分。
- 状态使用 badge，而不是大块彩色背景。
- 筛选使用 select、segmented control、radio group 或 checkbox。
- 数值设置使用 input、stepper 或 slider，按实际语义选择。
- 表格行需要 hover 状态。
- 所有可交互控件需要 focus-visible 状态。
- loading、empty、error 状态必须完整。

## 文案原则

- 短、准、面向任务。
- 避免营销语气。
- 避免解释 UI 本身。
- 中文界面优先使用安全运营语境：
  - 待审
  - 确认
  - 误报
  - 涉密
  - 安全
  - 忽略
  - 监控

## 页面要求

### 结果页

- 首屏必须能看到统计、筛选和结果列表。
- 过滤条件应紧凑排列。
- 表格列优先展示规则、仓库、文件、语言、上传时间、状态和操作。
- 风险状态必须清晰，但不要整页泛红。

### 详情页

- 左侧或顶部展示仓库、文件、规则和上传时间。
- 处理表单应靠近风险上下文。
- 代码查看区域可读性优先，长内容不能撑破布局。
- 受影响资产表格应便于复制和排查。

### 设置页

- 设置页应偏工具化，不做装饰。
- 页签应稳定，避免跳动。
- 每个设置分区都要有 loading、empty、error 状态。
- 敏感信息只显示必要的脱敏字段。

## 禁止模式

- 不做营销 landing。
- 不做大 hero。
- 不做渐变背景或装饰光斑。
- 不做玻璃拟态。
- 不做卡片套卡片。
- 不把表格做成大量独立卡片。
- 不使用大圆角胶囊作为主要按钮风格。
- 不使用纯图标且无 tooltip 的不熟悉操作。
- 不在页面中写“这个功能可以...”之类说明文。

## 设计 Token

实现时优先把这些 token 映射到 Tailwind 配置和 shadcn/ui 主题变量。

### 间距

使用 4px 基准：

- `4px`
- `8px`
- `12px`
- `16px`
- `20px`
- `24px`
- `32px`
- `40px`

规则：

- 紧凑控件内部间距优先使用 `8px` 或 `12px`。
- 页面区块间距优先使用 `24px` 或 `32px`。
- 不使用随意值，除非是为了对齐第三方组件或图表。

### 圆角

- 输入框、按钮、badge：`4px`
- 面板、表格容器、弹窗：`6px`
- 大型容器最大：`8px`

规则：

- 不使用大圆角胶囊作为默认风格。
- 状态 badge 可以使用较大圆角，但必须保持克制。

### 字体

优先使用系统字体栈：

```css
font-family: Inter, ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```

中文场景依赖系统字体回退即可，不额外引入字体文件。

### 字号

- 页面标题：`20px` 到 `24px`，字重 `600`
- 分区标题：`16px` 到 `18px`，字重 `600`
- 表格正文：`13px` 到 `14px`，字重 `400`
- 表单标签：`13px` 到 `14px`，字重 `500`
- 辅助信息：`12px` 到 `13px`，字重 `400`

规则：

- 不使用 viewport width 缩放字号。
- 字间距保持默认，不使用负 letter-spacing。
- 后台紧凑区域不得使用 hero 级字号。

### 颜色 Token

基础色：

- `background`: `#ffffff`
- `surface`: `#ffffff`
- `surface-subtle`: `#f6f8fa`
- `border`: `#d1d9e0`
- `divider`: `#d1d9e0`

文本色：

- `text-primary`: `#1f2328`
- `text-secondary`: `#59636e`
- `text-muted`: `#6e7781`
- `text-inverse`: `#ffffff`

语义色：

- `risk`: `#cf222e`
- `risk-bg`: `#ffebe9`
- `warning`: `#bf8700`
- `warning-bg`: `#fff8c5`
- `safe`: `#1a7f37`
- `safe-bg`: `#dafbe1`
- `info`: `#0969da`
- `info-bg`: `#ddf4ff`

交互色：

- `focus-ring`: `#0969da`
- `hover-surface`: `#818b981a`
- `selected-surface`: `#818b9826`

### 阴影

默认少用阴影，优先使用边框表达层级。

- 面板阴影：无
- 弹窗阴影：轻微阴影
- 下拉菜单阴影：轻微阴影

建议：

```css
box-shadow: 0 8px 24px rgba(16, 24, 40, 0.12);
```

仅用于弹窗、菜单和浮层。

### 密度

本项目是安全运营后台，默认采用中高密度布局。

- 表格行高：`40px` 到 `48px`
- 工具栏高度：`40px` 到 `48px`
- 表单控件高度：`32px` 到 `40px`
- 统计卡片高度：`88px` 到 `120px`

### 图标

- 优先使用 lucide-react 图标。
- 图标尺寸默认 `16px` 或 `18px`。
- 工具按钮可以只用图标，但必须有 tooltip 或 aria-label。
- 不手写 SVG 图标，除非没有合适图标或是业务专属图形。

## 可访问性要求

shadcn/ui 和 Radix UI 提供基础能力，但页面实现仍需要遵守以下规则。

### 基本原则

- 所有交互都必须支持键盘访问。
- 颜色不能作为唯一状态表达方式。
- 表单字段必须有关联 label。
- 错误信息必须靠近对应字段。
- focus-visible 状态必须清晰可见。
- 不牺牲可访问性来换取视觉效果。

### 表单

- 每个输入框、select、textarea、radio group 都必须有关联 label。
- 必填字段应有清晰标识。
- 校验错误应显示在字段附近。
- placeholder 不能替代 label。
- 保存中状态应禁用重复提交。

### 按钮和操作

- 纯图标按钮必须有 `aria-label` 或 tooltip。
- 危险操作必须有明确文案或视觉提示。
- disabled 状态必须可见。
- loading 状态不能只靠颜色表达。

### 表格

- 表头必须清晰。
- 行操作必须可键盘触达。
- 表格空状态必须可理解。
- 长文本截断时，应提供完整内容查看方式，例如 tooltip、详情页或复制操作。

### 弹窗和浮层

- Dialog 必须有 title。
- Dialog 打开后焦点应进入弹窗。
- Dialog 关闭后焦点应回到触发元素。
- 下拉菜单和弹出层必须支持键盘导航。

### 状态表达

风险、警告、安全等状态应同时使用：

- 文案
- 颜色
- badge 或图标

不要只靠红色、绿色区分状态。

### 对比度

- 正文和背景应保持足够对比度。
- 次级文本不能过浅。
- focus ring 不能被背景吞掉。
- 风险和安全状态在浅色背景上必须清晰可读。

### 检查方式

每个页面至少检查：

- Tab 顺序是否合理。
- Enter/Space 是否能触发按钮或选项。
- 表单 label 是否可见。
- 错误信息是否明确。
- 纯图标按钮是否有辅助名称。
- Dialog 或菜单是否能用键盘打开和关闭。
